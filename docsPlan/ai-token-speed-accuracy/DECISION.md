## 1. 결정 사항

### A. 토큰 부족 → "짐 줄이기" 3단계

1. **[단기] 경량 안전장치** — 프롬프트 조립 시점 토큰량 추정(글자수 근사) → 한도 근접 시 경고 + Ollama 유도 → 초과 시 가장 오래된 턴부터 단순 탈락(요약 압축 ❌ — 아래 3에서 버려질 코드라 과투자 금지).
2. **[단기] 진단 전문 가지치기** — FULL_DETAIL_DIAGNOSES=6 고정을 동적으로: 대화가 진행될수록(후보가 좁아질수록) 전문 개수를 줄이고, 배제된 진단은 명단에서 제거. 회수한 토큰 = 답변 공간.
3. **[중기·최종 그림] 임상 추론 루프 재설계** — "상태·채점은 JS, LLM은 얇게". 진단별 점수표를 JS 객체로 관리(기존 `main.js` `diagnosisScore`/`groupSatisfied` 재사용), 매 호출은 "현재 입력 + 확인할 기준 몇 개"만 전달 → 히스토리 누적 문제가 구조적으로 소멸. 별도 트랙으로 `docsPlan/`에 기획부터.

**기각(실측 근거)**: nPredict 상향(이미 1800, "올려도 잘림" 실측 `ai.js:275`), n_ctx 8192(GPU 드라이버 행 + CPU 수십 분 실측 `ai.js:33,836`), 모델 교체(Qwen3-4B AMD WebGPU 크래시 이력 + 카나나 한국어 특화).

### B. 속도 → 모델 교체 없이 2가지

1. **coi-serviceworker 도입** — GitHub Pages는 `_headers` 무시 → crossOriginIsolated=false → SharedArrayBuffer 불가 → wllama 싱글스레드(실측 확정). coi-serviceworker로 COI를 켜면 멀티스레드 wasm 활성화 가능. **도입 시 필수 테스트**: HF 모델 다운로드(1.4GB)·jsDelivr가 COI 아래에서 살아남는지 실기기 확인 + wllama가 실제 멀티스레드 빌드를 선택하는지 콘솔 확인.
2. **A의 토큰 다이어트가 곧 속도 개선** — CPU 추론에서 prefill 시간은 프롬프트 길이에 비례. 프롬프트가 절반이면 첫 글자 나오기까지의 대기도 크게 준다.

### C. 정확도 → "LLM은 얇게" + 그라운딩 강화

1. **그라운딩 유지·강화**(이미 구현됨) — data.js 기준 전문을 프롬프트에 주입, 모델 기억이 아니라 눈앞 자료로 판단. C의 모든 개선은 이 위에 쌓는다.
2. **임상 추론 루프에서 LLM의 일은 딱 3개** — ① 자유 서술에서 증상 추출 → data.js 항목 ID로 매핑 ② 다음 확인할 기준을 자연스러운 질문 문장으로 변환 ③ 최종 후보의 근거 설명. 점수 계산·가설 유지·후보 갱신은 전부 결정론적 JS.
3. **추출 매핑 신뢰성 확보(성패의 핵심)** — 출력 JSON 강제 + 코드가 실존 항목 키인지 검증(불일치 폐기) + wllama의 GBNF 문법 강제 샘플링 지원 여부 확인(되면 형식 깨짐 원천 차단).
4. **감별 질문 자동 선택** — 남은 유력 후보들을 가장 잘 갈라내는 기준을 다음 질문으로(기준 겹침 비교 코드로 자동화). 자동 채점에서 빠져 있는 `duration`(기간)·`other`(배제 기준) 필드를 질문 우선 소재로.
5. **출력 형식 제한** — 섹션별 개별 요청(기구현) 위에 "유력 진단 최대 3개, 근거 각 2줄 이내" 식 길이 제한 강화. 임상 참고 도구엔 장황함보다 구조화가 낫다.
6. **회귀 테스트 10~20케이스** — 입력 소견 → 기대 진단 후보. 채점은 결정론적이라 테스트 대상은 LLM 추출 단계. 기획과 병행해 케이스부터 축적.
7. **주의 2가지(설계 제약)** — 질문 루프를 심문처럼 만들지 않기(자유 서술이 주 통로, AI 질문은 제안 수준) / 공존진단 대비 복수 후보를 끝까지 유지(단일 수렴 ❌).

## 3. 실패 시나리오와 플랜 B

각 트랙의 확실성 등급과, 효과 없을 때의 대안. **착수 전에 성공 판정 기준(측정법)을 정해두는 것이 핵심** — "효과 없음"을 감으로 판단하지 않는다.

| 트랙 | 확실성 | 성공 판정 기준 | 안 되면 (플랜 B) |
|---|---|---|---|
| 1. 응급처치 (한도 가드+가지치기) | 높음 — 단, 크래시 원인이 문맥 한도라는 것은 아직 **추정**(정황 증거, HANDOFF 참고) | 크래시 재현 시나리오(장시간 대화)에서 ABORT 0회 + 답변 잘림 감소 | 가드를 넣었는데도 크래시 → 원인 가설이 틀렸다는 **증거 확보**가 성과. 그때는 콘솔 로그 채집으로 진짜 원인 추적(디버그 트랙 전환). 확실한 최후 수단 = Ollama 경로(이미 구현됨, 큰 문맥·큰 모델 가능) |
| 2. 임상 추론 루프 | 중간 — 성패는 2.1B 모델의 증상 추출 정확도에 달림 (본 문서 C-3) | 회귀 케이스 10~20개에서 추출 정확도 측정 — **전면 전환 전에 추출 단계만 먼저 파일럿** | ① LLM 역할 더 축소: 기존 키워드 매칭을 추출기로, LLM은 질문 문장화·설명만 ② LLM 제안 + 키워드 교차 확인 하이브리드 ③ 추출만 Ollama(큰 모델)에 위임하는 옵션 ④ 최악: 체크리스트 수동 유지 + AI는 질문·설명 보조만 — 어느 경우든 점수판-JS 구조 자체는 유효 |
| 3. coi-serviceworker | 중간 — COI가 켜져도 속도 향상 폭은 실측해야 (메모리 대역폭이 병목이면 코어를 늘려도 이득 제한) | 도입 전후 tokens/sec 비교 + 모델 다운로드 정상 + wllama 멀티스레드 빌드 선택 확인 | 스크립트 한 줄이라 **제거하면 즉시 원상복구** — 매몰 비용 거의 0. 효과 미미하면 속도는 1번의 프롬프트 다이어트 효과에 의존 |

공통 안전판: 순서가 "싼 것 먼저"라 큰 공사(2번)는 파일럿 측정을 통과해야만 본격 착수 — 실패해도 잃는 것이 기획 문서와 테스트 케이스뿐이며, 이는 어떤 대안에서도 재사용된다.

## 4. 근거 요약 (한 줄씩)

- 잘림·크래시의 원인은 출력 상한이 아니라 문맥 예산 — nPredict를 올려도 잘린다는 실측이 이미 코드 주석에 있다.
- 공책(4096칸)을 못 키우니(8192 실측 기각) 문제를 짧게 쓰는 것만 남는다.
- 작은 모델은 기억·계산이 약하다 — 계산은 코드가, 모델은 읽고 쓰는 일만 시켜야 정확해진다.
- 속도는 일꾼 수(스레드)와 읽을 양(프롬프트 길이)의 문제 — 둘 다 모델 교체 없이 줄일 수 있다.
