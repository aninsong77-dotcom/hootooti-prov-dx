# HANDOFF — 카나나 "(ABORT)" 크래시 & AI 선택 모달 이슈

> 상태: **원인 확정(직접 증거 확보), 조치 미착수(대기 중)**. 다음에 이 작업을 이어갈 때는 이 문서를 먼저 읽을 것.
> 마지막 갱신: 2026-07-13 (세션 중 기록 — 데스크탑 스크린샷으로 원인 확정)

## 1. 증상 (사용자 관찰)

- 노트북(AMD Radeon 660M 내장 GPU)과 **고사양 데스크톱 둘 다에서** 채팅 도중
  카나나(브라우저 내 로컬 AI) 사용 시 `이번 턴에서 오류가 발생했습니다: (ABORT)`
  오류가 발생.
- 두 사례 모두 **대화가 꽤 진행된 뒤**(핵심요약 → 유력가설 → 확인필요항목까지
  나온 뒤), 상담자가 확인필요항목에 대한 **긴 추가 답변**을 입력해서 전송한
  직후 발생. (스크린샷: `스크린샷 2026-07-13 141804.png`)
- 사용자 본인 추측: "토큰 부족 때문 아닐까, 아니면 내 추가 답변 때문일 수도"
  — 결론적으로 **거의 맞음** (아래 §2 참고, 다만 "토큰 부족"과는 다른 종류의
  한도임을 구분해서 설명해야 함).
- **고사양 컴에서 "AI 선택" 버튼을 눌렀을 때 화면이 노트북과 달랐다** + **Ollama
  다운로드 링크가 안 눌렸다**는 추가 제보 → **스크린샷 확보 후 원인 확인
  완료(아래 §2-3)**.
- **[2026-07-13 추가— 결정적 증거 확보]** 데스크탑 스크린샷 하단에 다음
  오류 문구가 그대로 찍혀 있었음:
  > `request (4438 tokens) exceeds the available context size (4096 tokens), try increasing it`

  이건 llama.cpp(Ollama·wllama 둘 다의 공통 엔진)이 문맥 한도 초과 시 내는
  **표준 오류 메시지**다. §2-2에서 "정황 증거 기반 추정"이라고 적어뒀던
  원인이 **이제 직접 증거로 확정됨** — 실제로 그 시점 요청이 4438토큰으로
  한도(4096)를 342토큰 초과했다.

## 2. 원인 분석

### 2-1. "(ABORT)" 자체의 정체 (이전 세션에 이미 확정됨)
`js/ai.js`가 쓰는 `@wllama/wllama`(카나나를 브라우저에서 돌리는 WebAssembly
라이브러리)는 내부 WASM 런타임이 치명적 오류로 죽으면 `Module.onAbort`가
발동되어 `(ABORT) ...` 형식의 메시지를 던진다(라이브러리 소스
`WLLAMA_EMSCRIPTEN_CODE`의 `onAbort`/`handleError` 확인함). 이건 우리 코드가
만든 문구가 아니라 **wllama 라이브러리가 강제로 붙이는 접두사**다.

### 2-2. 근본 원인 — 문맥 한도(N_CTX) 초과 [확정]
`js/ai.js`에 이미 예전부터 있던 주석(§ `N_CTX=4096 컨텍스트 한도`)이 정확히
이 상황을 경고하고 있었다:

> "요약 압축 없이 전체 누적 대화를 그대로 전달하므로, 대화가 매우 길어지면
> 한도 초과로 요청 자체가 실패할 수 있다"

- 카나나(wllama)는 `N_CTX = 4096`(js/ai.js 상수)으로 고정 운용 중 — 이건
  "이번 응답에 쓸 수 있는 토큰"(num_predict, 섹션별로 이미 작게 나눠둠, 이건
  해결됨)과는 **다른 한도**로, "지금까지의 대화 전체 + 이번 요청의 시스템
  프롬프트를 합친 총량"의 상한이다.
- 대화가 여러 턴 진행되고(핵심요약→유력가설→확인필요항목 세 번의 왕복) +
  상담자가 긴 추가 답변을 입력하면, 이 누적 총량이 4096을 넘어설 수 있다.
- **고사양 컴에서도 똑같이 났다는 게 중요한 단서였고, 위 §1의 실측 오류
  문구(4438 > 4096)로 확정됨** — GPU 성능·드라이버 문제(AMD 660M 한정
  이슈)가 아니라 하드웨어 성능과 무관한 **"대화 길이가 한도를 넘으면 실패"**
  임이 직접 증거로 확인됐다. "(ABORT)" 크래시와 "exceeds the available
  context size" 정상 오류는 **같은 근본 원인(문맥 초과)이 상황에 따라
  다르게 표출된 것**으로 보인다(wllama는 WASM 크래시로, Ollama 서버는
  깔끔한 오류 메시지로 — 표출 방식만 다를 뿐 트리거는 동일).
- Ollama 경로는 이 문제가 없다 — `num_ctx`를 8192로 이미 올려뒀고
  (`js/ai.js`의 `OLLAMA_MODELS[...].numCtx`), 애초에 한도 초과 시 처리 방식이
  wllama처럼 WASM 크래시로 죽는 게 아니라 서버가 오류를 정상적으로 반환하는
  구조라 훨씬 안전하다.

### 2-3. AI 선택 모달 차이 [확인 완료] + Ollama 링크 문제 [원인 미상, 재현 대기]
노트북·데스크톱 스크린샷 2장을 직접 비교해 확인함:
- **모달 모양 차이 → 정상 동작, 버그 아님.** 노트북은 Ollama가 감지된 상태라
  "모델 선택" 화면이, 데스크톱은 Ollama가 감지되지 않아 "설치 안내" 화면이
  나왔을 뿐 — 두 스크린샷의 "현재 상태" 문구가 그 차이를 그대로 보여줌.
  `js/ai-ui.js`의 `renderPopoverContent()`가 `state==='connected'` 여부로
  분기하도록 원래 그렇게 설계돼 있음. 코드 수정 불필요.
- **Ollama 다운로드 링크(`<a href="https://ollama.com/download" target="_blank">`)
  가 안 눌린 것 → 원인 미상, 코드 결함 가능성은 낮음.** 데스크톱 스크린샷에
  빈 "새 탭"이 떠 있는 게 보임 — 탭은 열렸으나 ollama.com으로 이동하지
  못한 것으로 보임. 앵커 태그 자체는 표준적(`target="_blank" rel="noopener"`)
  이라 코드 버그일 가능성은 낮고, 그 컴퓨터의 브라우저·보안 소프트웨어(팝업
  차단, 백신, 방화벽 등)가 원인일 가능성이 더 높음. 사용자에게 주소를 직접
  입력하거나 링크를 우클릭→새 탭에서 열기로 재시도해보라고 안내했으나,
  **재시도 결과는 아직 확인받지 못함.**

## 3. 이미 해결된 것 (참고 — 이번 이슈와 헷갈리지 말 것)
아래는 **과거 세션에서 이미 고친 별개 문제들**이다. 이번 "(ABORT)"와는 다른
버그였으니 다시 건드릴 필요 없음:
- `c69aa93` — 6단계를 한 응답에 다 담아 잘리던 문제 → 섹션별 개별 요청으로 분리
- `b188651` — 카나나에 요청을 몰아쳐서(6연발) GPU 부담으로 크래시 나던 문제
  → 사용자 클릭(다음 내용 보기) 페이싱으로 완화
- `6de3c34` — 진단 판단 불필요한 3단계(정리된소견·핵심요약·추가확인질문)는
  사전 텍스트 생략해 중복 프롬프트 처리 낭비 제거

**이번 "(ABORT)"는 이 조치들 이후에도 발생했다** — 즉 "응답 쪽 토큰"과
"요청 몰아치기"는 이미 고쳤는데도 나는 걸로 보아, §2-2의 **"입력(대화 누적)
쪽 문맥 한도"**가 진짜 원인일 가능성이 더 크다.

## 4. 다음에 시작할 때 할 일 (우선순위 순)

1. **카나나 대화 길이 안전장치 추가(가장 유력한 근본 조치)** — 매 요청 전에
   지금까지의 대화(conversation 배열) + 시스템 프롬프트를 합친 대략적인
   토큰/글자 수를 추정해서, `N_CTX`(4096)에 근접하면:
   - 카나나 사용을 더 진행하지 못하게 막고, "대화가 길어져 브라우저 AI
     한도에 가까워졌습니다. Ollama로 전환하거나 대화를 정리해 주세요"처럼
     **크래시가 나기 전에 먼저 경고**하는 방식이 유력한 후보.
   - 또는 오래된 turn을 요약·압축해서 문맥을 줄이는 방법도 있으나 훨씬
     손이 많이 감(정확도 트레이드오프도 고려해야 함) — 1차로는 "경고 후
     Ollama 유도"가 더 간단하고 안전한 선택지로 보임.
   - 실제 토큰 수를 정확히 재려면 토크나이저가 필요해 브라우저에서 정확한
     카운트는 어려움 — 글자 수 기반 대략치(예: 한국어 기준 1글자≈1.5~2토큰
     추정)로 근사하는 정도가 현실적. **이 근사치의 정확도도 실측 필요.**
2. Ollama 쪽은 num_ctx 8192로 이미 여유가 있어 우선순위 낮음(다만 대화가
   극단적으로 길어지면 8192도 넘을 수 있으니 동일한 안전장치를 Ollama
   경로에도 나중에 적용할지는 별도 판단).
