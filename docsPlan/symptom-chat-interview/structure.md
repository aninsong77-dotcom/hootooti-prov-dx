# structure.md — 증상 소견 채팅형 인터뷰 전환 구조 설계

작업명(제안): symptom-chat-interview
전제: 이전 초안에서 남겨두었던 미결정 항목(1~6)은 모두 확정되었습니다(`requirements.md` §2 참조). 구조는 확정된 원칙에 맞춰 재정리합니다. 단, §2.2의 자연어 종료 의도 감지 방식(구현 메커니즘) 등 일부는 impl 단계 구체화 대상으로 남습니다(`requirements.md` §4).

**구현 순서**: `engine-selector`를 먼저 구현하고, 이 트랙(`symptom-chat-interview`)은 그 이후 진행한다(파일 충돌 방지 목적, 확정). `js/ai.js`, `js/ai-ui.js`, `index.html`을 두 트랙이 공유하므로 `engine-selector` 병합 완료 후 착수한다.

**§2.5 재확정 반영**: 대화 보존 범위가 "새로고침 시 초기화"에서 "`sessionStorage`로 새로고침·페이지 왕복 후에도 유지"로 뒤집혔습니다(`requirements.md` §2.5). 이 문서의 §1(영향 파일), §2.1(상태 정의), §3(데이터 흐름), §4(요약표)를 그에 맞춰 갱신했습니다.

## 1. 영향 파일 전수 조사

| 파일 | 현재 역할 | 영향 |
|---|---|---|
| `index.html` | `#notes-input`(단일 textarea) + `#ai-result-card`(결과+1회성 후속질문 박스) + `#candidates-card`(키워드 후보, 별도) | **구조 변경** — 채팅 컨테이너로 교체, `#analyze-btn`/`#candidates-card`는 **제거**(`requirements.md` §2.4) |
| `js/ai-ui.js` | AI 분석 버튼 클릭 핸들러, 1회성 후속질문 렌더링, 모델 다운로드 진행률 표시, 엔진 안내 모달 | **전면 재작성** — 채팅 이벤트 루프로 전환. 엔진 안내 모달(`#ai-engine-info-btn` 등)은 그대로 유지 |
| `js/ai.js` | 프롬프트 빌더(`buildSystemPrompt`), Ollama/wllama 호출(`analyzeWithAI`), 후속질문 마커 파싱(`splitFollowUpSection`) | **수정** — 대화 배열 전체를 받아 매 턴 그대로 전달하도록 재설계(옵션 A 확정), 종료 판단은 자연어 응답 기반으로 전환 |
| `js/main.js` | `lastNoteText`/`lastCandidates`(모듈 스코프 상태), `saveResult()`, `clearNotes()`, 키워드 기반 `runDiagnosisAssist()` | **부분 수정 + 삭제** — `runDiagnosisAssist()`(키워드 기반 가진단) 및 관련 이벤트 바인딩 삭제, `lastCandidates` 상태 제거. `saveResult()`는 대화 배열의 "최종 정리된 결과"를 소스로 재구성 |
| `css/style.css` | `.assist-card`, `.candidates-card`, `.ai-followup-box` 등 기존 카드/박스 스타일 | **추가 + 정리** — 말풍선(채팅 버블) 스타일 신규 정의, `.candidates-card`(키워드 후보) 관련 스타일은 폐기 검토 |
| `js/data.js` | `DIAGNOSES` 정적 데이터(진단 기준 사전) | **영향 없음** — 프롬프트 그라운딩 소스로 그대로 재사용 |
| `checklist.html`, `icd11.html`, `about.html` | 별도 페이지, 이 채팅 UI 자체는 존재하지 않음 | **영향 없음(코드 수정 불필요)** — `sessionStorage`는 브라우저가 같은 origin의 모든 페이지에서 공유하므로, 이 페이지들은 대화 데이터를 **건드리지 않고 그대로 둔다**. 사용자가 이 페이지를 거쳐 `index.html`로 돌아왔을 때 `index.html`이 `sessionStorage`를 읽기만 하면 대화가 이어진다. 단, "가진단 받기" 제거 후 안내 문구에서 `checklist.html`로의 링크 강조는 여전히 필요 |

레이어 교차: **UI(index.html/css) + 이벤트 로직(ai-ui.js) + AI 연동/프롬프트(ai.js) + 전역 상태·저장(main.js)** 4개 레이어를 동시에 건드리므로 CLAUDE.md §1 기준 다중 레이어 변경이며, 기획 단계 필수 대상에 해당합니다.

## 2. 화면 설계 — 상태 정의

기존 "왼쪽 입력 / 오른쪽 결과" 2단 레이아웃(`.assist-layout` → `.assist-left` + `.assist-results-panel`)을 채팅형으로 어떻게 배치할지(좌우 분할 유지 vs 단일 컬럼 채팅창)는 시각 디자인 사안이라 별도 확인이 필요하지만, 최소한 아래 상태들은 빠짐없이 정의합니다.

### 2.1 채팅 컨테이너 상태

| 상태 | 설명 |
|---|---|
| 페이지 로드 시 복원 시도 | `DOMContentLoaded` 시점에 `sessionStorage`를 확인 — 저장된 대화가 있으면 [복원됨(대화 이어짐)]으로, 없거나 파싱 실패 시 [초기(빈 대화)]로 진입(§3.5 참조) |
| 초기(빈 대화) | 저장된 대화가 없거나(최초 방문·탭 재시작) 스키마 파싱에 실패한 경우. 안내 문구 + 첫 소견 입력 프롬프트만 표시. 기존 `.assist-desc`("관찰된 증상, 행동, 진술 등을 자유롭게 서술해 주세요.") 문구 자리를 대체. 키워드 기반 "가진단 받기" 버튼은 이 화면에서 완전히 제거되고, 대신 "정밀 체크리스트에서 직접 확인하기"류 안내 링크(`checklist.html`)로 대체 |
| 복원됨(대화 이어짐) | 새로고침 또는 `checklist.html`/`icd11.html`/`about.html` 왕복 후 돌아왔을 때, `sessionStorage`에 저장돼 있던 이전 대화 전체를 그대로 렌더링한 상태 — 신규 사용자 입장에서는 "초기"와 구분되는 상태이지만 UI 표현은 정상 진행 중인 대화와 동일 |
| 사용자 메시지 전송 중 | 입력창 잠금(중복 전송 방지), 전송 버튼 로딩 표시 |
| AI 응답 대기 중 | AI 말풍선 자리에 로딩 인디케이터(기존 `.ai-btn-spinner`/모델 다운로드 진행률 문구 로직을 계승) |
| AI 후속질문 말풍선 | 그 턴에서 AI가 확인이 필요하다고 판단한 질문 **전체를 한 번에** 하나의 AI 메시지로 표시(`requirements.md` §2.6 확정 — 통짜 텍스트, 개별 질문 UI 분리 없음) |
| 사용자 답변 말풍선 | 후속질문에 대한 자유 서술 답변. 답변 텍스트 자체가 자연어 종료 의도(예: "그만 물어봐줘")를 담을 수 있음(§2.2 종료조건과 직결) |
| 대화 순환 반복 | 위 4단계가 종료 조건 성립 전까지 **고정 라운드 없이** 반복 |
| 종료 트리거 — AI 자체 판단 | 가설연역적 판단 기준 충족(확인 불가 항목 소진) 또는 사용자 응답에서 종료 의도를 AI가 인식한 경우. 정확한 감지 메커니즘은 impl 단계 구체화 대상(`requirements.md` §4) |
| 종료 트리거 — 상담사 버튼 | 채팅 입력창 옆(또는 별도) "충분함" 버튼 클릭 — 대화 중 언제든 노출. AI 자체 판단과 경합 시 먼저 발생하는 쪽이 우선 |
| 최종 진단 표시 | 최종 후보 + 근거 텍스트를 별도 강조 말풍선 또는 하단 고정 결과 카드로 표시 — 채팅 흐름 안에 자연스럽게 이어지는 말풍선 형태를 유지할지, 기존 `#ai-result-card`처럼 별도 카드로 재분리할지는 시각 디자인 확인 필요(이 문서에서 확정하지 않음, 기능적 요건과 무관) |
| 오류 상태 | 기존 `statusEl.textContent = 'AI 분석 중 오류가 발생했습니다: ...'` 패턴을 채팅 말풍선 또는 인라인 배너 중 어느 형태로 표시할지 확인 필요. 최소 요건: 어느 시점(어느 턴)에서 실패했는지 사용자가 알 수 있어야 하며, 재시도 시 그 턴만 재전송 가능해야 함 |
| 빈 상태(오류 후 복구) | 오류로 중단된 대화를 이어갈지 처음부터 다시 시작할지 선택 가능해야 함 |
| 엔진 안내(기존 모달 계승) | 기존 `#ai-engine-info-btn` + `#ai-engine-modal-overlay`는 채팅 UI와 무관하게 그대로 유지(엔진이 Ollama/브라우저 중 무엇인지 안내) |

### 2.2 화면 전환 흐름

```
[페이지 로드 — sessionStorage 확인]
    ├─(저장된 대화 없음/파싱 실패)→ [빈 대화 상태]
    └─(저장된 대화 있음)→ [복원됨(대화 이어짐) — 마지막 상태 그대로 렌더링]

[빈 대화 상태]
    ↓ 사용자가 첫 메시지(소견) 입력·전송
[AI 응답 대기]
    ↓ 응답 도착 (매 turn 종료 시 conversation 스냅샷을 sessionStorage에 저장 — §3.5)
    ├─(AI가 확인할 게 있다고 판단)→ [AI 후속질문 말풍선 — 해당 턴 질문 전체 한 번에] → 사용자 답변 → [AI 응답 대기] (순환, 고정 라운드 없음)
    └─(AI가 충분하다고 판단(가설연역 기준 충족 또는 사용자 자연어 종료 의도 인식) OR 상담사가 "충분함" 버튼 클릭 — 먼저 오는 쪽)→ [최종 진단 표시]

(임의 시점에 새로고침 또는 checklist.html/icd11.html/about.html로 이동했다가 index.html로 복귀)
    → [페이지 로드 — sessionStorage 확인] → [복원됨(대화 이어짐)]으로 진행 중이던 지점부터 이어감

[최종 진단 표시]
    ↓ "결과 저장" 클릭
[.txt 다운로드 — 대화 전체가 아닌 "최종 정리된 결과"만 포함]

(탭을 닫거나 브라우저를 종료하면 sessionStorage가 소실되어, 다음 방문 시 [빈 대화 상태]부터 시작)
```

오류 발생 시 임의 지점에서 [오류 상태]로 분기하며, 사용자 재시도 시 오류 직전 상태로 복귀합니다.

### 2.3 컴포넌트 계층(재사용 가능 단위 식별)

- `ChatContainer` — 전체 대화 목록을 감싸는 스크롤 영역 (신규, `index.html`의 `.candidates-card#ai-result-card` 자리를 대체)
- `ChatMessage` — 단일 말풍선(사용자/AI 구분 스타일) — 재사용 단위
  - `ChatMessage.AIFollowUp` — 후속질문 전체를 한 번에 담는 AI 메시지 변형(통짜 텍스트, 확정)
  - `ChatMessage.FinalDiagnosis` — 최종 진단 강조 변형
- `ChatInputBar` — 하단 고정 입력창 + 전송 버튼 + "충분함" 버튼 (신규, 기존 `.notes-actions` 버튼 그룹을 재구성). **입력창에서 Enter 키를 누르면 전송 버튼 클릭과 동일하게 즉시 전송된다(카카오톡·ChatGPT·Claude 채팅창과 동일한 UX, `requirements.md` §2.1 확정).** 기존 `#ai-analyze-btn`("AI 분석" 버튼)은 별도 컴포넌트로 존치하지 않고 이 전송 동작에 완전히 흡수되어 **제거**된다.
- `EngineStatusBadge` — 현재 엔진(Ollama/브라우저) 표시 배지 — 기존 `#ai-mode-badge` 계승
- `EngineInfoModal` — 기존 `#ai-engine-modal-overlay` 그대로 재사용(변경 없음)

키워드 기반 "가진단 받기"(`#analyze-btn`, `#candidates-card`)는 컴포넌트 트리에서 **완전히 제외**합니다(제거 확정, `requirements.md` §2.4). `js/main.js`의 `runDiagnosisAssist()`, `renderCandidates()`, `formatCandidatesAsText()` 등 관련 함수도 impl 단계에서 삭제 대상으로 식별해야 합니다.

## 3. 데이터 흐름 설계

### 3.1 상태 모델(신규)

현재 `lastNoteText`(문자열 1개), `lastCandidates`(배열, 제거 예정) 두 개의 평면 변수로 관리되던 상태를, 채팅형에서는 아래 형태의 순서 있는 대화 배열로 관리합니다(정확한 필드 구성은 구현 단계에서 확정, 여기서는 개념 구조만 제시).

```
conversation = [
  { role: 'user', text: '(최초 소견 또는 후속답변)' },
  { role: 'assistant', text: '(AI 응답 본문)', followUp: [...] | null },
  ...
]
```

- `role`: Ollama `/api/chat`의 `messages` 배열과 동일한 `system`/`user`/`assistant` 개념을 그대로 재사용한다(옵션 A 확정 — §3.2 참조).
- 종료 여부는 별도 구조화 필드(`sufficient: bool`)로 저장하지 않고, AI 응답 텍스트 자체에서 매 턴 판단한다(마커/JSON 필드 방식 미채택 — §2.2, §3.3).
- `lastNoteText`는 `conversation[0].text`로 대체되며, 별도 모듈 변수로 이중 관리하지 않는다.
- `conversation` 배열은 메모리 상의 단일 진실 소스이면서 동시에 `sessionStorage`에 직렬화되는 대상이다(§3.5).

### 3.2 요청 조립 지점 (확정 — 옵션 A)

`js/ai.js`의 `analyzeWithAI(noteText, onProgress, followUpAnswerText)` 시그니처는 "소견 1개 + 답변 1개"만 받는 구조라 다회 순환을 표현할 수 없습니다. **대화 전체 누적 전달(옵션 A)로 확정**되었으므로, 매 호출 시 `conversation` 배열 전체를 `messages: [...]`에 그대로 실어 전달하는 구조로 재설계합니다. Ollama `/api/chat`, wllama `createChatCompletion` 모두 이미 `messages` 배열을 받는 인터페이스이므로 API 자체의 변경 폭은 작습니다.

**리스크(그대로 인지하고 진행)**: `N_CTX=4096`(`js/ai.js:26`)이라는 컨텍스트 한도를 요약 압축 없이 계속 소모하므로, 대화가 길어지면 한도를 초과할 수 있습니다. 이 리스크는 `requirements.md` §5 "테스트 필수 대상"에 명시했으며, 초과 시 처리 방식(오래된 턴 자르기, 사용자 안내 등)은 impl 단계에서 별도로 설계합니다.

`buildSystemPrompt()`의 `isFollowUp` 이분법(1차/2차 두 단계만 존재, `js/ai.js:121-173`)은 다회 순환 + 자연어 종료 판단을 지원하도록 확장이 필요합니다.

### 3.3 응답 파싱 지점 (확정 — 마커 방식 미채택)

`splitFollowUpSection()`(`js/ai.js:287-295`)의 고정 마커(`### 추가확인질문`, `(추가 질문 없음)`) 방식은 후속질문 분리에는 계속 쓸 수 있으나, **"충분함" 판단에는 이 방식을 쓰지 않기로 확정**했습니다. 종료 여부는 AI 응답의 자연어 내용 자체(및 사용자 최신 발화의 종료 의도)로 매 턴 판별하며, 구체적 판별 로직(별도 분류 프롬프트를 추가로 태울지, 시스템 프롬프트 지시로 흡수할지)은 `requirements.md` §4에 impl 단계 구체화 대상으로 명시되어 있습니다.

### 3.4 저장(`saveResult`) 데이터 소스 (확정)

`js/main.js:453`의 `$notesInput.value.trim() || lastNoteText.trim()` 보정 로직은 "단일 textarea 값이 사라져도 모듈 변수에 백업이 남아있다"는 전제로 작동합니다. 채팅형에서는 이 전제가 사라지므로, `conversation` 배열을 저장 시점의 단일 진실 소스(source of truth)로 삼습니다.

**저장 포맷 확정**: 대화 전체를 시간순으로 저장하지 않습니다. 기존 `saveResult()`와 동일한 개념 — "최초 소견 + 최종 정리된 진단 결과"만 추려 `.txt`로 저장합니다. 구체적으로는 `conversation[0]`(최초 소견)과 종료 시점 최종 AI 응답(최종 진단)을 조합하는 형태가 되며, 세부 텍스트 조립 규칙은 impl 단계에서 기존 `saveResult()` 포맷을 최대한 유지하는 방향으로 구현합니다.

### 3.5 sessionStorage 저장·복원 지점 (신규 — §2.5 재확정 반영)

**저장**: `conversation` 배열이 바뀔 때마다(사용자 메시지 추가, AI 응답 수신, 종료 처리 등 매 turn 경계) `JSON.stringify(conversation)`로 직렬화해 `sessionStorage.setItem(<key>, ...)`으로 저장한다. 키 이름은 다른 페이지의 스크립트와 충돌하지 않는 접두어(예: `houtoti-` 계열)를 impl 단계에서 정한다.

**복원**: `index.html` 로드 시(`DOMContentLoaded`) `sessionStorage.getItem(<key>)`을 조회한다.
- 값이 없으면(최초 방문, 또는 탭을 새로 열어 `sessionStorage`가 비어 있는 경우) [빈 대화 상태]로 진입한다.
- 값이 있으면 `JSON.parse()`로 역직렬화해 `conversation` 배열을 채우고, 저장된 메시지 순서 그대로 `ChatMessage` 컴포넌트들을 렌더링해 대화를 이어간다.

**스키마 방어(권고, `requirements.md` §4)**: 역직렬화 또는 구조 검증(예: 배열인지, 각 항목에 `role`/`text` 필드가 있는지)에 실패하면, 오류를 사용자에게 노출하지 않고 **조용히 [빈 대화 상태]로 새로 시작**하도록 설계할 것을 권고한다. 이는 이후 코드 변경으로 `conversation` 항목 구조(§3.1)가 바뀌었을 때, 이전 세션에서 저장된 값이 새 코드와 맞지 않아 깨지는 것을 방지하기 위함이다. 구체적 검증 로직·버전 필드 부여 여부는 impl 단계에서 정한다.

**용량 리스크(인지만 해둠, `requirements.md` §4)**: `sessionStorage`는 보통 도메인당 5~10MB 한도를 갖는다. §3.2에서 대화 전체를 요약 없이 누적하기로 확정했으므로, 매우 긴 대화에서는 이 한도에 이론적으로 근접할 수 있다. 이번 범위에서 심각하게 다룰 문제는 아니라고 보되, impl 단계에서 이 가능성을 인지하고 있어야 한다.

**다른 페이지와의 관계**: `checklist.html`/`icd11.html`/`about.html`은 이 채팅 UI 자체가 없으므로 별도 JS 수정이 필요 없다(§1). `sessionStorage`는 브라우저가 같은 origin의 모든 탭·페이지에 공유하는 저장소이므로, 사용자가 이 페이지들을 거쳐 `index.html`로 돌아오기만 하면 위 복원 로직이 자동으로 대화를 이어 렌더링한다.
