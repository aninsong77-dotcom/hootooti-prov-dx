## 1. 배경 — 현재 코드베이스 상태 (파일 재확인 완료)

이번 전환 대상은 "증상 소견 입력 → AI 후속질문 → 상담사 답변 → AI 진단" 흐름이며, 현재는 1회성 질의응답 구조로 구현되어 있습니다.

- `index.html:59-72` — `#notes-input`(`textarea`, `rows=10`)에 "내담자 증상 소견"을 입력하고, `#ai-analyze-btn`을 누르면 AI 분석이 시작된다. 소견 입력창은 AI 분석과 키워드 기반 "가진단 받기"(`#analyze-btn` → `#candidates-card`) 두 기능이 함께 쓰고 있다. **(§2.4에서 후자는 이번 전환 시 제거로 확정)**
- `index.html:81-96` — `#ai-result-card` 안에 결과 텍스트(`#ai-result-text`)와 1회성 후속질문 박스(`#ai-followup-box`)가 있다. 후속질문 박스는 질문 목록(`#ai-followup-questions`, 텍스트 통짜) + 답변 입력 `textarea`(`#ai-followup-answer`) + "답변 반영해서 다시 분석" 버튼(`#ai-followup-submit-btn`)으로 구성되며, 제출 시 2차 분석 1회만 수행하고 종료된다(추가 순환 없음).
- `js/ai-ui.js:79-87` `showFollowUp()` — 후속질문 텍스트를 그대로 통째로 렌더링(개별 질문 파싱 없음). `js/ai-ui.js:76` `lastNoteText` 변수 하나로 원본 소견을 기억해 2차 분석 시 재사용.
- `js/ai.js:110-173` `buildSystemPrompt()` — 가설연역적 추론 프롬프트. `isFollowUp` 플래그로 1차(후속질문 생성 포함)/2차(최종 결론만) 두 모드만 존재. 마커 문자열 `### 추가확인질문`(`FOLLOWUP_MARKER`)로 후속질문 섹션을 구분.
- `js/ai.js:297-323` `analyzeWithAI(noteText, onProgress, followUpAnswerText)` — 매 호출마다 `noteText` + (있다면) `followUpAnswerText` 하나만으로 새 system/user 메시지 쌍을 만들어 단발 요청을 보낸다. 대화 히스토리를 누적 전달하는 구조가 아니다. **(§2.7에서 전체 히스토리 누적 전달로 확정 — 재설계 대상)**
- `js/ai.js:175-232` — Ollama 연동(`OLLAMA_URL`, `OLLAMA_MODEL='qwen3:4b-instruct'`, `detectOllama()` 15초 타임아웃, `/api/chat`, `num_predict:1800`). 미감지 시 브라우저 내 wllama(Kanana GGUF) 폴백(`ensureModelLoaded`).
- `js/main.js:43-44` `lastCandidates`, `lastNoteText` — 모듈 스코프 변수로만 상태 유지(새로고침 시 소실, DB·localStorage 없음). **(§2.5에서 채팅 대화는 `sessionStorage`로 새로고침·페이지 왕복 후에도 유지하도록 확정 변경 — 서버 DB 없음 원칙 자체는 그대로 유지)**
- `js/main.js:443-486` `saveResult()` — 소견 원문 + AI 분석 결과 텍스트 + 키워드 기반 후보 목록을 합쳐 `.txt` 파일로 다운로드. 452-453행에서 `$notesInput.value.trim() || lastNoteText.trim()`로 소견을 반드시 포함(과거 "저장 시 소견 누락" 버그 수정 지점). **(§2.3에서 "최종 정리된 결과만 저장"이라는 현재 개념을 유지하기로 확정 — 소스만 대화형 흐름의 최종 산출물로 교체)**

## 2. 확정된 요구사항

### 2.1 UI 형태 (사용자 확인 완료 — 최초 확정, 입력 방식 추가 확정)
완전 채팅형(말풍선 대화 목록, Claude 채팅창과 같은 구조)으로 **전면 교체**한다. 기존 `textarea` + 결과 카드(`#ai-result-card` 내부의 1회성 후속질문 박스) 구조는 갈아엎으며 부분 유지하지 않는다. 최초 소견 입력도 "대화의 첫 메시지"로 취급한다(별도 입력창 + 결과창 분리 구조 폐지).

**입력·전송 방식(확정)**: 카카오톡·ChatGPT·Claude 채팅창과 동일하게, 하단 입력창에 내용을 타이핑하고 **Enter 키(또는 전송 버튼 클릭)**로 바로 전송한다. `#ai-analyze-btn`("AI 분석" 버튼)은 **제거**하고, 그 역할(첫 소견 전송 → 분석 시작 트리거)은 `ChatInputBar`의 전송 동작이 그대로 흡수한다 — 별도의 "분석 시작" 버튼을 추가로 두지 않는다.

### 2.2 대화 종료 조건 (사용자 확인 완료 — 최초 확정, 신호 방식은 추가 확정)
AI가 스스로 "충분하다"고 판단하면 종료 가능, 또는 상담사가 "충분함" 버튼을 직접 눌러도 종료된다. **둘 중 먼저 발생하는 쪽이 우선**하여 최종 진단 단계로 전환시킨다.

**AI 자체 판단 기준(확정, impl 단계에서 신호 형식만 구체화):**
- 기존 가설연역적 추론 구조(`buildSystemPrompt()`)의 판단 기준을 그대로 유지한다 — "확인 불가능한 항목이 더는 없다"고 AI가 스스로 판단하면 종료.
- **추가로**, 사용자가 채팅 중 자연어로 "이제 진단을 내려달라"는 뉘앙스를 표현하면 그것도 충분함 신호로 인식한다(예: "그만 물어보고 결론 내줘", "이 정도면 충분한 것 같아요" 등 자유 서술 — 고정 문구가 아님).
- 특수 마커 토큰(`### xxx` 같은 구조화 텍스트) 방식은 채택하지 않는다 — 사용자가 "텍스트 마커가 뭔지 모르겠다"고 답한 것을 근거로, 마커 대신 **AI가 자연어 응답 안에서 종료 의도를 판단**하는 방식으로 확정.
- **impl 단계에서 구체화가 필요한 부분**: 사용자의 자연어 종료 의도를 정확히 무엇으로 감지할지(예: 매 사용자 턴마다 별도 분류 프롬프트를 태울지, AI 응답 프롬프트 자체에 "사용자가 종료를 원하면 이렇게 응답하라"는 지시를 심어 응답 자체에서 판별할지 등)는 이번 기획에서 확정하지 않는다. 프롬프트 설계와 함께 별도로 구체화한다.

### 2.3 결과 저장 (해소됨 — 확정)
"결과 저장" 버튼은 채팅형 전환 후에도 계속 존재한다. **대화 전체를 저장하지 않고, 현재 `saveResult()`와 같은 개념으로 "최종 정리된 결과"만 저장한다** — 소스만 단일 `textarea`/`lastCandidates`에서 대화형 흐름의 최종 산출물(최초 소견 + 최종 진단 결과)로 바뀔 뿐, 저장 포맷의 기본 철학(참고용 가진단 결과 요약본)은 유지한다.

### 2.4 가진단 받기 버튼 (해소됨 — 확정)
키워드 기반 "가진단 받기" 버튼(`#analyze-btn`, `#candidates-card`, 비-AI 기능)은 이번 전환 시 **제거**한다. 사용자가 이 기능을 쓰고 싶으면 별도 페이지(`checklist.html` 등 정밀 체크리스트)로 직접 이동해 수동으로 확인하도록 변경한다.

### 2.5 대화 보존 범위 (재확정 — 이전 결정 뒤집힘)
사용자가 명시적으로 요청: "결과가 계속 쌓이면서(ChatGPT/Claude 채팅창처럼) 이어지길 원하고, 페이지에 머무는 동안은 물론 다른 페이지(`checklist.html`, `icd11.html`, `about.html`)를 다녀와도 대화가 지워지지 않았으면 좋겠다. 사이트(탭)를 끄지 않는 한 유지되길 원한다." 이 사이트는 SPA가 아니라 페이지마다 완전히 새로 로드되는 멀티페이지 정적 사이트임을 사용자에게 설명했고, 해결책으로 **브라우저 `sessionStorage`**를 제안해 동의를 받았다.

**확정 내용**: 대화는 `sessionStorage`에 저장되어 **새로고침·다른 페이지(`checklist.html`/`icd11.html`/`about.html`) 왕복 후에도 유지된다.** 브라우저 탭을 닫거나 사이트를 완전히 벗어나면(`sessionStorage` 특성상) 소실된다. `localStorage`(탭을 닫아도 영구 보존)는 채택하지 않는다 — 사용자가 명시한 조건("사이트를 끄지 않는 한")과 정확히 일치하는 것은 `sessionStorage`이지 `localStorage`가 아니다.

이전 초안에서 "새로고침 시 초기화, 영속화 도입 없음"으로 확정했던 내용은 이 항목으로 **대체**된다.

### 2.6 후속질문 표시 방식 (해소됨 — 확정)
후속질문은 한 번에 전부(그 턴에서 AI가 확인이 필요하다고 판단한 항목 전체) 보여준다. 사용자 답변을 본 뒤 AI가 추가로 확인할 게 있다고 판단하면 대화를 계속 이어간다 — **고정된 라운드 수(예: "1차·2차만") 없이**, §2.2의 종료 조건(AI 자체 판단 또는 상담사 "충분함" 버튼, 먼저 오는 쪽)이 성립할 때까지 반복될 수 있다.

### 2.7 대화 히스토리 전달 방식 (해소됨 — 확정)
AI에 매 턴 전달하는 컨텍스트는 **대화 전체를 누적한 그대로**(ChatGPT/Claude 같은 구조) 전달한다. 요약 압축은 하지 않는다. `js/ai.js`의 `analyzeWithAI()`가 소견 1개+답변 1개만 받는 현재 시그니처는 대화 배열 전체를 받는 구조로 재설계되어야 한다(구체적 구현은 impl 단계).

### 2.8 진단 프롬프트 보강 (여전히 별도 범위 — 변경 없음)
방향 미확정. `js/ai.js`의 현재 프롬프트(`buildSystemPrompt`, §1 참조)를 함께 보면서 별도 세션에서 정하기로 함. 예시로 거론된 방향: 진단기준 상세화, 감별진단 로직 강화, 사례 데이터 반영. **본 기획 문서는 이 항목을 확정하지 않는다.**

## 4. 확정된 원칙 중 impl 단계 구체화가 필요한 부분

아래 항목들은 원칙은 확정되었고 **구체적 구현 방식만** impl 단계에서 정합니다(임의로 결론내지 않음).

- **사용자 자연어 종료 의도 감지 방식**(§2.2): "그만 물어봐 줘" 류의 자연어 표현을 AI가 어떤 메커니즘으로 판별해 종료 신호로 연결할지(응답 프롬프트 내 지시로 흡수 vs 별도 분류 단계 추가 등)는 이번 기획에서 확정하지 않는다.
- **`sessionStorage` 용량 한도 초과 가능성**(§2.5): 브라우저 `sessionStorage`는 보통 도메인당 5~10MB 한도를 가진다. 대화가 매우 길어지면(§2.7 요약 압축 없이 전체 누적) 이 한도에 근접할 가능성이 이론적으로 존재한다. 이번 범위에서 심각하게 고려할 문제는 아니라고 판단하나, impl 단계에서 이 가능성을 인지하고 있어야 한다(구체적 방어 로직 설계는 이번 기획에서 확정하지 않음).
- **저장된 대화의 스키마(형식) 버전 문제**(§2.5): `sessionStorage`에 저장한 `conversation` 배열의 필드 구성이 이후 코드 변경으로 바뀌면, 이전 세션에서 저장된 값과 새 코드가 기대하는 구조가 맞지 않을 수 있다. **권고**: 저장된 값을 읽어 파싱할 때 실패하거나 예상 구조와 다르면, 오류를 띄우지 않고 조용히 새 대화로 시작하도록 방어적으로 설계할 것을 권고한다. 구체적 스키마 버전 관리 방식(버전 필드 부여 여부 등)은 이번 기획에서 확정하지 않는다.

## 5. 테스트 필수 대상

이 기획 단계에서 식별한, 개발 단계에서 테스트 우선순위를 두어야 할 로직입니다.

- **대화 종료 조건 판정 로직**: AI 자체 판단(가설연역적 기준 충족 + 자연어 종료 의도 인식) 신호와 상담사 "충분함" 버튼 중 먼저 발생하는 쪽이 실제로 우선 적용되는지(동시 발생·경합 상황 포함).
- **자연어 종료 의도 오탐/미탐**: 사용자가 종료를 의도하지 않았는데 AI가 종료로 오판하거나, 반대로 명확히 종료를 원했는데 AI가 놓치는 경우를 최소화하는지 — 마커 방식보다 자유도가 높은 만큼 리스크가 크므로 우선순위 높음.
- **대화 히스토리 누적 시 컨텍스트 한도(`N_CTX=4096`, `js/ai.js:26`) 초과 처리**: 대화가 길어질 때 오류 없이 처리(잘림/한도 초과 안내 등)되는지 — 요약 압축을 하지 않기로 확정했으므로 이 리스크가 실제로 존재함.
- **결과 저장(`saveResult` 계열) 데이터 무결성**: "최종 정리된 결과"가 실제로 빠짐없이 `.txt`에 포함되는지(과거 "저장 시 소견 누락" 버그 재발 방지 — §1 `js/main.js:453` 보정 로직의 대체 지점).
- **Ollama/wllama 엔진 전환 시 대화 상태 일관성**: `currentEngine()` 판정이 바뀌는 시점(브라우저 세션 중 Ollama가 켜지거나 꺼지는 경우)에도 진행 중인 채팅 히스토리가 깨지지 않는지.
- **가진단 받기 버튼 제거 후 회귀 확인**: `#analyze-btn`/`#candidates-card` 제거가 `js/main.js`의 다른 로직(`runDiagnosisAssist`, `clearNotes` 등)에 부작용을 남기지 않는지.
- **`sessionStorage` 저장·복원 정확성**: 새로고침, `checklist.html`/`icd11.html`/`about.html` 왕복 후 대화가 실제로 그대로 이어지는지(순서·역할·내용 손실 없음). 탭을 닫았다 다시 열면(새 탭) 대화가 소실되는지도 함께 확인.
- **스키마 불일치 시 방어적 복구**: 저장된 `sessionStorage` 값의 구조가 코드 기대와 다를 때(예: 이전 버전 데이터, 손상된 값) 오류 없이 조용히 새 대화로 시작하는지.
