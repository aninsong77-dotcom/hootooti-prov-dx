## 1. 배경 — 현재 코드베이스 상태 (파일 재확인 완료)

- `js/ai.js:179-183` — Ollama 모델은 `OLLAMA_MODEL = 'qwen3:4b-instruct'`로 고정되어 있다. 코드 주석에 "qwen3:4b(생각모드 버전)는 최종 답을 내기 전 영어로 아주 길게 '생각'하다가 토큰 한도를 다 써버려 실사용이 어려움 — 바로 한국어 최종 답만 내는 instruct(생각모드 없는) 버전을 쓴다"고 명시되어 있어, 현재 고정은 의도적 선택이었다.
- Ollama 라이브러리에는 `qwen3:4b-thinking`(및 양자화 버전 `qwen3:4b-thinking-2507-q4_K_M`) 태그가 실존한다 — 현재 쓰는 `qwen3:4b-instruct`의 생각과정 버전 짝. (조사로 확인된 사실 — 가정 아님)
- `js/ai.js:226` — Ollama 호출 시 `num_predict: 1800`(코드 주석: "1200에서도 최종 결론 직전에 잘리는 경우가 실측되어 1800으로 상향").
- `js/ai.js:318` — 브라우저 wllama 호출 시 `max_tokens: 1800`.
- 사용자가 이전에 `thinking` 버전을 테스트했다가 "내용이 너무 많아서 다 볼 수 없어서" 제외한 이력이 있다. 원인 조사 결과, CSS 쪽(`overflow-y:auto`)은 이미 스크롤 가능하도록 되어 있어 잘림·과다 노출의 원인이 아닌 것으로 확인되었고, 위 `num_predict`/`max_tokens` 1800 한도가 원인으로 추정된다. (추정임을 명시 — 실제 원인은 impl 단계에서 실제 모델로 검증 필요)
- `index.html:83` — `<h2>AI 분석 결과 (Kanana)</h2>`가 **하드코딩**되어 있다. Ollama(qwen3)로 분석해도 제목은 항상 "(Kanana)"로 표시된다. 바로 옆 `#ai-mode-badge`는 `js/ai-ui.js:157`, `js/ai-ui.js:191`에서 `currentEngine()`으로 정확히 갱신되는데, `<h2>` 제목만 동적 갱신 로직이 없다. (버그로 확인됨)
- `js/main.js:461` — 결과 저장 텍스트에도 `'[AI 분석 결과 (Kanana, 브라우저 내 로컬 AI)]'`가 **하드코딩**되어 있어, 저장 파일(`saveResult()`)에도 실제 사용 엔진과 무관하게 항상 "Kanana"로 찍힌다. (버그로 확인됨)
- `index.html:63-71` — `.notes-actions` 안에 `#analyze-btn`, `#ai-analyze-btn`, `#clear-notes-btn`, `#save-result-btn`이 `.btn.btn-compact` 클래스로 나란히 배치되어 있다. `css/style.css:294` `.btn-compact` 클래스 존재 확인.
- `index.html:73`, `index.html:122-151` — 기존 "AI 엔진 안내" 모달(`#ai-engine-info-btn` → `#ai-engine-modal-overlay`)이 이미 존재하며, "2단계 · 더 빠르게 쓰고 싶다면" 섹션에 `ollama.com/download` 링크와 `ollama pull qwen3:4b-instruct` 안내가 포함되어 있다.
- `js/ai.js:187-202` `detectOllama()` — `/api/tags`를 15초 타임아웃으로 호출해 모델 목록을 확인하고, `models.some(...)`으로 대상 모델 존재 여부를 판정한다. **`try`/`catch` 구조에서 모든 실패 원인(모델명 불일치, 네트워크 오류, 권한 거부, 기타 예외 등)이 `catch (e) { ollamaAvailable = false; }`(199행) 한 줄로 뭉뚱그려지며, 실패 사유를 구분하지도 사용자에게 알리지도 않는다.** (코드로 확인된 사실 — 가정 아님)
- `js/ai.js:281-283` `currentEngine()` — `ollamaAvailable === true`면 `'ollama'`, 아니면 `'browser'`(wllama) 반환.

## 2. 이 티켓의 목적

"결과 저장" 버튼 옆에 AI 엔진(생각과정 없음/있음)을 선택할 수 있는 버튼을 추가하고, 동시에 현재 엔진 표시가 실제 엔진과 무관하게 "Kanana"로 고정 출력되는 버그를 함께 수정한다. 또한 이번 논의 중 발견된 `detectOllama()`의 실패 원인 은폐 구조를 개선해, 사용자가 왜 Ollama 연동이 안 되는지 알 수 있게 한다. 같은 화면 영역(엔진 표시·감지)에 걸쳐 있는 문제이므로 하나의 티켓으로 묶는다.

## 3. 확정된 요구사항

### 3.1 신규 기능 — 생각과정 유무 선택 버튼 (확정)
- "결과 저장" 버튼과 같은 스타일(`.btn.btn-compact`)로, 그 옆에 새 버튼을 배치한다.
- 버튼 클릭 시 두 옵션을 노출한다:
  - **생각과정 없음 (현재 기본)** — 지금의 `qwen3:4b-instruct`
  - **생각과정 있음 (qwen3:4b-thinking)** — 신규
- 이미 받아둔(pull된) 모델은 어느 쪽인지 표시한다.
- 각 옵션에 마우스 오버 시 툴팁으로 설명을 보여준다(무엇을 다운받는지, 무엇이 다른지).

**채택 UI 방향(B안, 확정)**: 버튼 1개 → 클릭 시 위 두 옵션이 노출되는 방식. A안(버튼 2개를 처음부터 나란히 배치)과 B안을 놓고 논의하다 B안을 추천했고, **사용자가 "네, B안으로 확정"이라고 명시적으로 확인했다.**

### 3.2 Ollama 미감지 환경에서의 버튼 처리 (해소됨 — 안 2 확정)
버튼은 계속 보이되 **비활성화(회색) 처리**한다. 호버/클릭 시 "Ollama가 필요합니다" 툴팁을 보여주며, **이 안내는 새로 작성하지 않고 기존 "AI 엔진 안내" 모달(`#ai-engine-info-btn` → `#ai-engine-modal-overlay`, `index.html:73, 122-151`)로 연결**한다 — 이미 `ollama.com/download` 링크와 설치 안내가 있으므로 문구를 중복 작성하지 않는다.

### 3.3 자동 다운로드(`/api/pull` 직접 트리거) (변경 확정 — 기존 "제외"에서 뒤집힘)
`qwen3:4b-thinking`을 이미 pull하지 않은 상태에서 사용자가 옵션을 선택하면, **앱이 Ollama의 `/api/pull` 엔드포인트를 직접 호출해 다운로드를 트리거**한다. 기존처럼 사용자가 터미널에서 `ollama pull`을 수동 실행하도록 요구하지 않는다.

### 3.4 다운로드 용량 확인 문구 (해소됨 — 보여줌으로 확정)
자동 다운로드(§3.3)를 실제로 트리거하기 **직전**에, 수 GB 단위 용량에 대한 확인 문구를 노출한다. 정확한 문구·용량 수치는 impl 단계에서 실측 후 확정한다.

### 3.5 토큰 한도 조정
`qwen3:4b-thinking` 사용 시 생각 과정 자체가 토큰을 많이 소비하므로, 현재 두 곳의 토큰 한도를 올려야 한다는 요구사항으로 반영한다:
- `js/ai.js:226` `num_predict: 1800` (Ollama 호출)
- `js/ai.js:318` `max_tokens: 1800` (브라우저 wllama 호출)

목표 범위는 예시로 4000~6000이 거론되었으나, **정확한 값은 확정하지 않는다** — thinking 모델을 실제로 붙여 응답이 잘리지 않는지 impl 단계에서 검증 후 정한다. wllama(브라우저 내 Kanana)는 이번 신규 옵션과 무관한 별도 엔진이므로, `max_tokens` 조정이 wllama에도 적용되어야 하는지는 impl 단계에서 판단한다.

### 3.6 버그 수정 — 엔진 표시 하드코딩
같은 화면 영역이므로 이 티켓 범위에 포함한다:
- `index.html:83` `<h2>AI 분석 결과 (Kanana)</h2>` → `currentEngine()` 기반으로 동적 표시되도록 수정(`#ai-mode-badge`가 이미 하는 방식과 동일한 패턴 재사용).
- `js/main.js:461` `'[AI 분석 결과 (Kanana, 브라우저 내 로컬 AI)]'` → 저장 시점의 실제 엔진을 반영하도록 수정.

### 3.7 (신규) Ollama 감지 실패 이유 사용자 안내
`detectOllama()`(§1 참조)가 실패했을 때, 조용히 Kanana(wllama)로 넘어가지 않고 **실패 사유를 사용자가 이해할 수 있는 말로 보여준다**. 예: "Ollama에 연결할 수 없습니다", "필요한 모델이 설치되어 있지 않습니다" 등. 정확한 문구·분류 방식은 impl 단계에서 `catch (e)`의 `e` 내용을 실제로 확인하며 설계하되, 최소한 아래 두 가지는 구분하는 것을 목표로 한다:
- 연결 자체가 안 됨(fetch 실패·타임아웃 등 네트워크 레벨)
- 연결은 됐지만 필요한 모델이 목록에 없음(모델명 불일치·미설치)

### 3.8 (신규) 디버그 정보 표시
§3.7의 사람이 읽는 실패 안내와 별도로, **원본 에러 정보(에러 메시지, 어떤 단계에서 실패했는지)를 확인할 수 있는 디버그용 버튼 또는 "자세히 보기" 토글**을 추가한다. 목적: 사용자가 이 내용을 복사해 다음 세션에 붙여넣으면, 개발자(Claude)가 원격으로 원인을 훨씬 정확히 진단할 수 있게 하기 위함. UI 형태(별도 버튼 vs 실패 안내 안의 접이식 상세보기)는 impl 단계에서 정한다.

### 3.9 배경 — 실제로 겪은 미제 사례 (원인 불명, 단정하지 않음)
이번 논의 계기가 된 사용자 사례: 고사양 컴퓨터에서 Ollama 설치 + `ollama pull qwen3:4b-instruct`를 명령어 그대로 정확히 복사해 실행했음에도 앱이 계속 Kanana(브라우저 wllama)로만 작동했고, 브라우저의 사설망 접근 허용 팝업도 뜨지 않았다. 모델명 오타, 팝업 거부 모두 원인에서 배제되었으나, **정확한 원인은 확인되지 않았다 — 단정하지 않고 미제로 기록한다.** §3.7·§3.8의 실패 원인 안내·디버그 정보 기능이 이런 사례를 다음에 더 정확히 진단하기 위한 목적으로 추가되었다.

## 5. 남은 미결정 항목 (Open Question — 임의로 결론내지 않음)

1. **`qwen3:4b-thinking` `/api/pull` 진행 상태를 프로그레스바로 보여줄지.**
   §3.3에서 앱이 `/api/pull`을 직접 호출하는 자동 다운로드로 확정되었으므로, 이제는 가정이 아니라 실제로 필요한 설계 결정이 되었다. 현재 `ensureModelLoaded()`(wllama, 브라우저 다운로드)는 `onProgress` 콜백으로 다운로드 진행률을 받는 구조가 이미 있다(`js/ai.js:245-274`, `js/ai-ui.js`의 `formatBytes`/`formatRemaining` 진행률 UI). Ollama의 `/api/pull`도 스트리밍 진행률 응답을 주는지는 이번 조사에서 확인하지 않았으며, **구현 단계에서 실제 API를 붙여보며 확인이 필요**하다.

## 6. 테스트 필수 대상

- **엔진 표시 정확성**: `<h2>` 제목·저장 텍스트가 실제 `currentEngine()` 값과 항상 일치하는지 — 특히 세션 중 Ollama 감지 상태가 바뀌는 경우.
- **thinking 모델 응답 잘림 여부**: 상향된 토큰 한도에서 `qwen3:4b-thinking`의 생각 과정 + 최종 답변이 실제로 끝까지 나오는지(현재 1800에서 잘렸던 문제의 재발 여부).
- **Ollama 미감지 시 버튼 비활성화 동작**: 비활성화 상태에서 호버/클릭 시 기존 엔진 안내 모달이 정확히 뜨는지.
- **`/api/pull` 자동 다운로드 전체 경로**: 트리거 → 용량 확인 문구 → (동의 시) 다운로드 → 완료 후 모델 전환까지 끊김 없이 이어지는지, 실패 시(네트워크 끊김 등) 사용자에게 적절히 알려지는지.
- **실패 원인 분류 정확성**: `detectOllama()`가 "연결 자체 실패" vs "모델 없음"을 실제로 구분해 사용자에게 보여주는지 — 오분류 시 사용자가 잘못된 조치(예: 이미 설치된 모델을 다시 설치)를 하게 될 위험이 있으므로 우선순위 높음.
- **디버그 정보의 실용성**: 표시된 원본 에러 정보만으로 실제 장애 상황(§3.9의 미제 사례 같은 것)을 다음 세션에서 재현·진단할 수 있는 수준인지.
